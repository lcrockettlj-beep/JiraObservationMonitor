JIRA OBSERVATION MONITOR
PYTHON PROJECT MANUAL
=====================

Purpose
-------
This manual explains Python as it is used in the Jira Observation Monitor project.
It is NOT a generic Python textbook.
It is a practical guide for a developer or maintainer who needs to understand:

- what Python does in this project,
- which Python files matter most,
- how the files work together,
- how data moves through the backend,
- how to add or change functionality safely,
- how to diagnose and fix common Python-side problems.


1. WHAT PYTHON DOES IN THIS PROJECT
-----------------------------------
In Jira Observation Monitor, Python is responsible for:

1. Running the web application
2. Loading and processing CSV / JSON inputs
3. Building the combined operational data model
4. Calculating estate, site, user, and project metrics
5. Building drilldown datasets
6. Supplying the HTML templates with data
7. Serving raw data through an API endpoint

Python does NOT currently:
- render charts in the browser
- manage client-side interactivity beyond supplying HTML/JS/CSS
- push outbound alerts
- modify Atlassian data

The application is read-only.

3. HOW PYTHON FILES WORK TOGETHER
---------------------------------
The Python backend currently follows this flow:

A. Flask route is called in web.py
B. web.py rebuilds the operational payload
C. web.py calls multiple service modules
D. service modules return data and/or drilldowns
E. web.py merges everything into one payload
F. web.py renders a template or returns JSON

In simple terms:

request -> web.py -> service modules -> unified data payload -> template/API response


4. THE MAIN BACKEND PATTERN
---------------------------
The project uses a central orchestration pattern.
The most important backend idea is:

ONE route layer + MANY helper/service modules + ONE combined data payload

The current route layer is in web.py.
The service logic is split into domain modules.
The combined data payload is built in web.py and reused by multiple routes.

This is good because:
- one place controls page data
- multiple templates can reuse the same values
- drilldowns are centralised

This is risky because:
- if orchestration breaks, multiple pages can break at once
- web.py currently carries a lot of responsibility


5. FLASK BASICS IN THIS PROJECT
-------------------------------
This project uses Flask as the Python web framework.

Core pattern from web.py:

    app = Flask(
        __name__,
        template_folder=str(BASE_DIR / "templates"),
        static_folder=str(BASE_DIR / "static"),
    )

Meaning:
- HTML templates are expected in templates/
- CSS/JS files are expected in static/


5.1 Route example
-----------------
A simple route pattern looks like this:

    @app.route("/")
    def home():
        data = _build_data()
        return render_template("home.html", **_common_template_data(data))

What this means:
- When someone visits /, Flask runs the home() function
- Python builds the data payload
- Python renders home.html using that payload


5.2 Other active routes

6. THE MOST IMPORTANT FUNCTION: _build_data()
----------------------------------------------
The most important Python function in the current backend is _build_data().

Why it matters:
- homepage depends on it
- estate page depends on it
- reference page depends on it
- detail pages depend on it
- /api/data depends on it

If _build_data() fails, the platform is effectively down.


6.1 What _build_data() does
---------------------------
High-level steps:

1. Load latest CSV files
2. Build estate metrics
3. Merge project counts
4. Merge project intelligence
5. Merge change detection
6. Merge site discovery
7. Attach billing summary
8. Build a single drilldowns dictionary
9. Return one combined data object


6.2 Example core snippet
------------------------
This is the kind of orchestration pattern used in web.py:

    users_rows, users_file = _load_csv_rows("export-users*.csv")
    managed_rows, managed_file = _load_csv_rows("*managed_accounts*.csv")

    data = build_estate_metrics(users_rows, managed_rows)

    data["sites"] = _merge_project_counts(data.get("sites", []))

    project_intelligence = load_project_intelligence_from_latest_run()
    data["project_intelligence"] = project_intelligence
    data["sites"] = _merge_project_intelligence(data.get("sites", []), project_intelligence)

    change_detection = load_latest_run_change_detection()
    data["change_detection"] = change_detection
    data["sites"] = _merge_change_detection(data.get("sites", []), change_detection)

    site_discovery = load_site_discovery_from_latest_run()
    data["site_discovery"] = site_discovery

This is the backbone of the system.


7. DATA INPUTS AND LOADING
--------------------------
Python currently reads two important CSV data sources directly.

7.1 Users export
----------------
Loaded using a file pattern like:

    export-users*.csv

7.2 Managed accounts export
---------------------------
Loaded using a file pattern like:

    *managed_accounts*.csv

The helper function _load_csv_rows(pattern) in web.py:
- finds files matching the pattern
- picks the newest one
- reads rows using csv.DictReader
- returns the rows and source filename

Example snippet:

    def _load_csv_rows(pattern):
        matches = glob.glob(str(BASE_DIR / pattern))
        if not matches:
            return [], None

        latest_file = max(matches, key=os.path.getmtime)
        with open(latest_file, newline="", encoding="utf-8-sig") as f:
            rows = list(csv.DictReader(f))
            return rows, os.path.basename(latest_file)

Important note:
If filenames don’t match the expected pattern, the app may silently miss the latest data.


8. DRILLDOWN ENGINE
-------------------
The drilldown system is one of the most important Python concepts in this project.

The app has many links such as:
- /detail/change::summary
- /detail/discovery::summary
- /detail/project::summary
- /detail/site::<site>
- /detail/access::<product>

These only work because web.py assembles a combined dictionary:

    data["drilldowns"]

Each drilldown entry contains things like:
- title
- reason
- atlassian_area
- columns
- rows


8.1 Detail route pattern
------------------------
The dynamic detail route looks like this:

    @app.route("/detail/<path:key>")
    def detail(key: str):
        data = _build_data()
        item = data.get("drilldowns", {}).get(key)
        if not item:
            abort(404)
        ...

Meaning:
- key is the part after /detail/
- Python looks for it in the drilldowns dict
- if missing, the page returns 404

This means broken or renamed drilldown keys are a very common source of failure.


8.2 How drilldowns are merged
-----------------------------
web.py currently merges drilldowns from several services.
Typical pattern:

    drilldowns = data.get("drilldowns", {})
    drilldowns.update(billing.get("drilldowns", {}))
    drilldowns.update(build_change_detection_drilldowns(change_detection))
    drilldowns.update(build_project_drilldowns_from_latest_run())
    drilldowns.update(build_site_discovery_drilldowns(site_discovery))
    data["drilldowns"] = drilldowns

Important meaning:
- multiple modules contribute into one shared namespace
- duplicate keys would overwrite each other
- a missing update() call could remove access to an entire drilldown domain


9. SITE ENRICHMENT FUNCTIONS
----------------------------
web.py also contains merge/enrichment helpers. These enrich site cards after the base estate metrics are created.

Examples:
- _merge_project_counts(sites)
- _merge_project_intelligence(sites, project_intelligence)
- _merge_change_detection(sites, change_detection)

These functions take the existing list of sites and add fields to each site dict.

Example pattern:

    for site in sites:
        site_key = site.get("site")
        project_data = project_counts.get(site_key, {})
        site["project_count"] = project_data.get("project_count")
        site["issue_count_total"] = project_data.get("issue_count_total")

This is how a site's displayed metrics grow over time inside the application.


10. SORTING LOGIC IN PYTHON
---------------------------
Detail pages use Python sorting logic when the user clicks column headers.

Current backend sorting concepts:
- request.args gets sort and order values
- _sort_rows sorts drilldown rows
- last-seen style fields are treated specially as dates

Important support helpers in web.py include:
- _parse_possible_datetime(value)
- _is_last_seen_field(sort_key)
- _sort_rows(rows, sort_key, order)

Why this matters:
A developer changing detail page fields should know that not all columns are treated equally.
Columns such as last_seen or last_active may be sorted as dates instead of text.

Example pattern:

    sort_key = request.args.get("sort", "").strip()
    order = request.args.get("order", "").strip().lower()
    rows = _sort_rows(rows, sort_key, order)


11. COMMON PYTHON PATTERNS USED IN THIS PROJECT
-----------------------------------------------
11.1 Dictionaries everywhere
----------------------------
A lot of project data is represented as Python dictionaries.

Example:
- one site = dict
- one drilldown = dict
- entire payload = dict

This means developers should be comfortable with:
- .get()
- nested dicts
- lists of dicts

Example:

    site_name = site.get("site_name")
    discovery_summary = site_discovery.get("summary", {})


11.2 Lists of dicts
-------------------
Many views rely on lists containing rows as dictionaries.

Example:

    rows = [
        {"Name": "Alice", "Email": "alice@example.com"},
        {"Name": "Bob", "Email": "bob@example.com"},
    ]

This is the core row format for drilldowns.


11.3 Defensive .get() access
----------------------------
The project often uses .get() instead of direct [] access to avoid crashes if a key is missing.

Example:

    value = row.get(col, "")

This is safer than:

    value = row[col]

because it avoids KeyError when the field is absent.


11.4 Filtering with list comprehensions
---------------------------------------
Site grouping uses Python list comprehensions.

Example:

    data["critical_sites"] = [s for s in data["sites"] if s.get("status") == "critical"]
    data["warning_sites"] = [s for s in data["sites"] if s.get("status") == "warning"]
    data["stable_sites"] = [s for s in data["sites"] if s.get("status") == "stable"]

This is how pages receive grouped site lists.


12. HOW TO ADD SOMETHING NEW SAFELY
-----------------------------------
12.1 Add a new page route
-------------------------
If you want to add a new page, the safe process is:

1. create a new template in templates/
2. create a new Flask route in web.py
3. reuse _common_template_data(data) if the page uses common dashboard values
4. test the new route directly in browser

Example route pattern:

    @app.route("/my-new-page")
    def my_new_page():
        data = _build_data()
        return render_template("my_new_page.html", **_common_template_data(data))


12.2 Add a new drilldown
------------------------
Safe process:

1. choose which service module owns the drilldown
2. add the drilldown data in that module
3. confirm web.py merges that drilldown source into data["drilldowns"]
4. add a link in a template
5. test the exact /detail/<key> URL

Example drilldown structure:

    drilldowns["estate::example"] = {
        "title": "Example Drilldown",
        "reason": "Why this matters",
        "atlassian_area": "Atlassian Administration",
        "columns": ["Name", "Email", "Site"],
        "rows": [
            {"Name": "Alice", "Email": "alice@example.com", "Site": "site-a"}
        ],
    }


12.3 Add a new site card metric
--------------------------------
Safe process:

1. decide which Python service owns the metric
2. populate that value into each site dict in the relevant merge helper
3. use it in templates/home.html and/or templates/estate.html
4. verify /api/data shows the field before editing the template

Good rule:
If the metric does not appear in /api/data, don’t try to use it in HTML yet.


13. PYTHON DEBUGGING WORKFLOW FOR THIS PROJECT
----------------------------------------------
When something breaks, use this exact order.

Step 1 — Can Python start?
--------------------------
Run:

    python web.py

If it fails before Flask starts:
- syntax error or import error is likely
- fix traceback first


Step 2 — Can Python build the payload?
--------------------------------------
Open:

    http://127.0.0.1:5000/api/data

If this fails, _build_data() or one of the services is failing.


Step 3 — Are templates the issue?
---------------------------------
If /api/data works but /estate fails:
- route or template issue is likely
- check render_template name and template file existence


Step 4 — Is it a drilldown-specific issue?
------------------------------------------
If pages work but a detail page 404s:
- drilldown key mismatch is likely

Check:
- template href value
- data["drilldowns"] keys


14. COMMON PYTHON ERRORS AND WHAT THEY MEAN HERE
------------------------------------------------
14.1 SyntaxError
----------------
Meaning:
- invalid Python syntax in a file

Usually caused by:
- missing colon
- broken indentation
- unclosed bracket/quote

Fix:
- open the file and go to exact traceback line


14.2 ImportError / ModuleNotFoundError
--------------------------------------
Meaning:
- Python can’t import a module

Possible causes:
- module file missing
- dependency not installed
- broken syntax inside imported module

Fix:
- try importing the module directly:

    python -c "import estate_metrics"


14.3 KeyError
-------------
Meaning:
- code tried to access a dict key that does not exist

In this project this often means:
- assumption about row structure is wrong
- service output shape changed

Safer pattern:

    value = row.get("field_name", "")


14.4 AttributeError
-------------------
Meaning:
- code called something like .get() on a value that isn’t a dict

In this project this often means:
- expected dict but got None or list
- service returned unexpected type

Fix:
- inspect variable shape
- print(type(variable)) temporarily in debug


14.5 404 on detail route
------------------------
Meaning:
- detail key not found in drilldowns map

Fix:
- inspect available keys in /api/data
- compare to template link exactly


15. SAFE PYTHON EDITING RULES FOR THIS PROJECT
----------------------------------------------
Rule 1:
Do not edit web.py, templates, and CSS all at once unless absolutely necessary.

Rule 2:
If changing one service file, verify /api/data before touching the front end.

Rule 3:
Always confirm the field exists in Python data before trying to show it in Jinja/HTML.

Rule 4:
Use Git commits for every stable checkpoint.

Rule 5:
If a change breaks the app, revert quickly before trying a second fix.


16. GIT WORKFLOW FOR PYTHON CHANGES
-----------------------------------
Recommended pattern:

    git add .
    git commit -m "Short meaningful message"
    git push

Examples:
- "Add estate registry route"
- "Fix drilldown key mismatch"
- "Merge project issue metrics into site cards"
- "Restore theme and nav layout"

If something goes wrong before commit:

    git checkout -- web.py

If everything is wrong and not committed:

    git checkout -- .

If already committed and you want to undo logically:

    git revert HEAD

18. WHAT NOT TO REFACTOR TOO EARLY
----------------------------------
Do NOT rush into:
- splitting every helper into many tiny files
- changing all dict structures
- renaming all drilldown keys
- redesigning _build_data() without tests

Why:
The current system is stable partly because web.py still centralises orchestration.
Premature refactor can add complexity without immediate value.


19. RECOMMENDED FUTURE PYTHON REFACTOR PATH
-------------------------------------------
Best next Python refactor direction later:

Phase A:
Move small utility helpers out of web.py
- _parse_possible_datetime
- _is_last_seen_field
- _sort_rows

Phase B:
Move orchestration logic into a dedicated dashboard service module
- _build_data
- _merge_project_counts
- _merge_project_intelligence
- _merge_change_detection

Phase C:
Keep web.py focused on routes only

This should only be done after current behaviour is fully understood.


20. QUICK PYTHON RECOVERY CHECKLIST
-----------------------------------
If the backend breaks, do this exact order:

1. Run:

    python web.py

2. Read the traceback.
3. If app starts, open:

    /api/data

4. If /api/data fails, inspect _build_data() and service imports.
5. If /api/data works but a page fails, inspect route/template pairing.
6. If pages work but drilldown fails, inspect key name.
7. If values are missing, inspect /api/data before touching templates.


21. ONE-PARAGRAPH SUMMARY
-------------------------
Python in Jira Observation Monitor is the operational engine of the system. It loads the latest Atlassian export data, builds a unified estate model, enriches it through multiple domain-specific service modules, supplies the templates with structured data, and powers all drilldowns and API output. The most important Python file is web.py, because it orchestrates the whole app. The most important maintenance rule is to make small isolated changes, verify /api/data immediately, and commit only stable states.
