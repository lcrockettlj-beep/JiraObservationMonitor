# 🧠 Jira Observation Monitor (JiraOps Platform)

## 🔷 Overview

The **Jira Observation Monitor** is a **read-only operational observability platform** for Atlassian/Jira environments.

It provides:

* estate-wide visibility
* site-level intelligence
* user/permission insights
* change detection
* drill-down exploration

The platform acts as a **live operational console**, enabling administrators to:

* quickly detect risk or anomalies
* understand usage and structure
* navigate to actionable areas in Atlassian

***

# 🎯 Core Purpose

The platform solves a key gap:

> Atlassian does not provide a single pane of glass for **cross-site operational intelligence and risk visibility**

This system provides:

* **centralised monitoring**
* **real-time summarisation**
* **drilldown intelligence**
* **action-driven insights**

***

# 🧱 Platform Architecture

## 🔹 Backend (Python / Flask)

* `web.py` → controller + routes + data assembly
* modular intelligence services:
  * `estate_metrics.py`
  * `project_counts.py`
  * `change_detection.py`
  * `site_discovery.py`
  * `billing_catalog.py`

### Responsibilities

* ingest CSV + JSON exports
* build unified data model
* calculate metrics + summaries
* provide drilldown datasets
* render templates

***

## 🔹 Frontend (Templates + Static)

* Jinja templates (`templates/`)
* CSS split:
  * `app.css` → shared design system
  * `home.css` → homepage layout
  * `detail.css` → drilldown tables
* JS:
  * `theme.js` → dark/light mode

***

## 🔹 Data Sources

### Current

* Managed accounts export (CSV)
* Users export (CSV)
* Operational snapshot (`latest_run.json`)

### Design note

System is built to support:

* additional Jira sites
* future APIs/webhooks
* snapshot comparisons

***

# 📊 Intelligence Layers

## 1️⃣ Estate Intelligence

From `estate_metrics.py`

Tracks:

* user states (active/inactive)
* access gaps
* product usage
* license distribution

### Examples:

* disabled accounts
* users without site access
* inactive users with no usage

***

## 2️⃣ Project Intelligence

From `project_counts.py`

Tracks:

* project counts per site
* sampling coverage
* change movement

### Purpose:

→ detect:

* project growth/decline
* abandoned estates
* imbalanced usage

***

## 3️⃣ Change Detection

From `change_detection.py`

Compares snapshots:

### Outputs:

* total change count
* user delta
* project delta
* site attention flags

### Purpose:

→ identify:

* sudden growth or drop
* structural changes
* anomalies over time

***

## 4️⃣ Site Discovery

From `site_discovery.py`

Builds:

* tracked sites
* excluded sites
* discovered estate

### Purpose:

→ maintain **accurate estate registry**

***

## 5️⃣ Billing & Reference Intelligence

From `billing_catalog.py`

Tracks:

* Atlassian apps
* marketplace apps
* product access coverage

### Purpose:

→ align:

* cost vs usage
* licensing footprint
* product exposure

***

# 🚨 Monitoring & Alert Strategy

The platform uses **implicit alerting**, not email/alerts yet.

## Current alerting model:

### Visual prioritisation:

* **Critical**
* **Warning**
* **Stable**

### Signal sources:

* user inactivity
* license utilisation
* zero activity sites
* change spikes
* access anomalies

***

## “Needs Attention” Cards

This is your **top-level alert system**

Examples:

* Managed Disabled Accounts
* No Tracked Jira Access
* Inactive Users Without Access
* Bitbucket-only users
* Excluded Sites

👉 These are **action-first signals**
(not noise, not metrics)
